# 讓 AI agent 連續跑 27 小時:/goal 功能與「Evaluation 才是關鍵」

**主題分類:** AI / Agentic Engineering(代理工程)
**來源:** YouTube〈我的 AI agent 連續跑了 27 個小時,/goal 功能怎麼用?〉(Gary Chen,2026-05-24,約 16 分;依繁中逐字稿整理)
**整理日期:** 2026-05-29

---

## 1. 核心命題:真正的自動化是解放「注意力」

把任務從你手上移到 AI 手上,只是自動化的前半段;後半段是把那件事 **從你心上徹底移開**。認知科學:只要心裡掛著一件沒處理完的事(「等 AI 跑完我要記得改那個按鈕」),就會偷偷吃掉你的腦力——而 **注意力才是這時代真正稀缺的東西**。作者上週讓 agent **連續跑 27 小時、無人介入**。

---

## 2. /goal:三家同時推出的同名功能

Claude Code、OpenAI Codex、Hermes Agent **幾乎同時推出名字一樣的 `/goal`**:在對話框打 `/goal` + 你要它達成的目標,它就 **自己跑到完成為止**(進階可設 token 上限)。

**它在解決什麼?AI 會偷懶。** 你一定遇過:做到一半停下來問「我可以繼續嗎?要 A 還 B?」,或更糟——沒做完卻說做完了,寫個漂亮總結把球丟回給你。

> **根因 = Context Anxiety(上下文焦慮)。** Anthropic 2025 年底研究:LLM 執行任務時會一邊看自己的 context window 用了多少,**一感覺快滿就開始慌、急著 wrap up 交差了事**——作者稱之為「下班心態」,是刻在 LLM 基因裡的惰性。`/goal` 就是為對抗這種惰性而生。

## 3. 運行原理:執行者 + 評審(胡蘿蔔吊在豬鼻子前)

```mermaid
flowchart LR
    G["你的 goal<br/>(完成定義)"] --> E["實作者 Executor<br/>產出東西"]
    E --> J["評審 Judge<br/>每輪檢查:目標達成了嗎?"]
    J -->|否,點出問題| E
    J -->|是| D["停止,交付"]
```

兩個角色協作:**實作者** 做髒活、**評審** 每輪檢查產出品質與進度;只要目標沒達成,評審就點出問題、叫實作者繼續。讓 agent **能自我鞭策** 跑到終點,你不用每三分鐘回來催它。

> 前身:去年紅過的插件 **Ralph Loop**(Ralph = 辛普森家庭裡有點傻但永不放棄的小孩);如今三家把這理念變成官方 feature。也與 [[karpathy-autoresearch]] 的「永不主動停的實驗迴圈」、[[ai-harness-explained]] 的「驗證迴圈」同源。

## 4. 怎麼寫 goal prompt:好壞差在「完成定義」

- **爛例:** 「把這個專案改得好一點。」→ 「好一點」沒有邊界,AI 做一兩個小改動、五分鐘說「我讓它變更好了」就收工,它自定的 definition of done 你不會滿意。
- **好例:** 「把結帳頁反應速度降到 **0.2 秒以內**,用速度測試工具 **驗證**;其他功能保持完好,**只能改結帳區塊的程式碼與相關測試**;每改一次 **記錄** 改了什麼、測出多少、下一步;若工具跑不起來或方法都試過了就 **停下來** 告訴我你試過什麼、卡在哪、需要什麼資訊。」

**好 goal prompt 的五個要素:**

| 要素 | 意義 | 例子 |
|---|---|---|
| ① Outcome | 完成後應是什麼狀態 | 反應速度 < 0.2 秒 |
| ② Verification | 怎麼證明真的完成 | 用速度測試工具驗證 |
| ③ Constraints | 哪些不能動 | 只能改結帳頁 |
| ④ Iteration policy | 每次嘗試之間做什麼 | 記錄改動/結果/下一步 |
| ⑤ Error handling | 何時暫停回報而非硬做 | 工具跑不起來就停下回報 |

> 關鍵不是 prompt 多會寫,而是 **你把「完成」定義得多明確**。定義得好,它就跑到完成、且是你要的樣子;定義得爛,AI 三分鐘草草收工、白燒 token。

## 5. 質化工作怎麼辦?把「漂亮」拆成可評分的維度

寫程式有明確標準(測試通過、速度 < 0.12s、lint 乾淨),但白領工作多是質化(圖好不好看?文章好不好?)。Anthropic 研究讓 Claude 設計漂亮網頁:AI 會把自己做的醜網站自評成「現代感、高質感」。解法是把「漂亮」拆成 **四個維度**:

1. **設計品質** — 顏色/字體/排版有沒有共同營造獨特氛圍與識別感。
2. **原創性** — 是刻意設計還是套預設模板(列出 AI 愛用元件,如「卡片+漸層色」太通用 → 判沒原創)。
3. **技術執行** — 字體階層、間距、配色、對比是否一致(每頁標題大小不一 → 不過)。
4. **可用性** — 使用者看得懂介面、找得到主按鈕、能達成目的。

**校正技巧:** 故意 **加重模型平常做不好的維度權重**(Claude 技術執行/可用性不錯,但設計品質/原創性平庸 → 把評分往這兩個弱項偏)。你的評分標準 = 校正模型預設行為的 benchmark。評審拿 rubric **用 Playwright 開瀏覽器、截圖、看使用者真正看到的畫面打分**(不是看程式碼)。

> **創意躍遷案例:** 做美術館網站,第 9 輪是「乾淨但你預期得到」的深色 landing page;**第 10 輪** 突然把網站重想成 **空間體驗**——CSS 透視渲染 3D 房間、黑白方格地板、藝術品掛牆、穿過虛擬門進不同展廳(「像 Claude 被梵谷附身」)。這種躍遷 **不可能從單次 prompt 產出、且非線性**(第 10 輪可能勝過第 15 輪),但只要評審與執行者持續對話,複雜度與野心會累積、偶爾迸出飛躍。

**結論:真正關鍵是 Evaluation——不是 prompt engineering、不是 context engineering,而是你能不能清楚定義「什麼叫做得好」並讓評審據以打分。** 這也是 [[karpathy-autoresearch]](val_bpb 當唯一指標)指向的同一件事。

## 6. 應用案例:把你的「品味」拆成 rubric 的 6 步 SOP

以「請 AI 寫作」為例:

1. **先讓 AI 做一輪(baseline):** 別急著寫標準,丟 5–10 個主題讓它隨便跑,測現有基準能力。
2. **逐篇親看、記下你皺眉的點與原因:** 開頭沒 hook?沒具體例子?——這些就是 rubric 雛形(如:又用「在這個快速變化的時代」、用一般人不用的成語、沒有具體人名/數字)。
3. **分類成幾個維度:** 50 條皺眉理由 → 收斂成 3 類(如「邏輯鬆散」「沒有人味」「開頭沒 hook」)——同 Anthropic 的四維度做法,只是換領域。
4. **每個維度寫成具體案例(核心):** 別寫「避免 AI 味」這種抽象描述,要寫「**絕對不要用破折號**」「**絕對不要用『不是 A,而是 B』句型**」。仿 Anthropic 的 front-end-design skill:「絕對不要用 Inter/Roboto/Arial/system fonts」「絕對不要用紫漸層蓋白卡片」——把 AI 一犯再犯的錯一個個點名。
5. **用多樣化案例取代單一描述:** Anthropic 血淚教訓——寫「博物館等級質感」結果全變博物館風;改成列 **11 種美學風格**(brutalist、art deco、pastel、industrial、retro-futuristic…)讓 Claude 視情況選一種。**AI 會 overfit 你給的範例,要寫多個方向確保多樣性。**
6. **把 rubric 餵給評審 agent、放進 goal prompt 跑:** 「這些是你的審核標準,沒達成前持續迭代不要停。」**初期務必人工檢查每輪產出**,確認評審判斷與你親眼看的一致;不一致代表 rubric 還沒抓到你的品味 → 回去改。跑三四輪後,你心裡對「做得好」的定義就被一條條梳理出來。

> **最深的一句:** 寫 evaluation rubric 表面是給 AI 用,實際是 **逼你把只存在腦袋裡的模糊品味寫成文字**;一旦寫成文字,AI 就能幫你 **守住它、大規模執行它**。這一刻你就從「AI 的協作者」變成「**定義自己品味的 AI 管理者**」。

**小練習:** 挑一件你最常做、最需要個人品味的任務(寫貼文、回 email、做行銷圖、剪片),花 30 分鐘跑這 6 步。
